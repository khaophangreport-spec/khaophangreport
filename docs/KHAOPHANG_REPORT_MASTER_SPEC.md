# Khaophang Report

## เอกสารกำหนดแนวทางพัฒนาเว็บแอประบบรับแจ้งและติดตามปัญหาชุมชนตำบลเขาพัง

**ชื่อระบบ:** Khaophang Report  
**ชื่อภาษาไทย:** ระบบรับแจ้งและติดตามปัญหาชุมชนตำบลเขาพัง  
**ประเภทระบบ:** Responsive Web Application  
**รูปแบบการออกแบบ:** Mobile-first  
**สถานะเอกสาร:** เอกสารตั้งต้นสำหรับเริ่มพัฒนา  
**เวอร์ชัน:** 1.0.0  

---

# 1. ภาพรวมโครงการ

Khaophang Report เป็นเว็บแอปสำหรับรับแจ้งปัญหา เหตุเดือดร้อน ความเสียหาย ข้อร้องเรียน และข้อเสนอแนะจากประชาชนในพื้นที่ตำบลเขาพัง โดยประชาชนสามารถส่งข้อมูลผ่านโทรศัพท์มือถือ แนบภาพ ระบุตำแหน่ง และติดตามผลการดำเนินงานได้ด้วยรหัสติดตามเรื่อง

ระบบต้องช่วยให้ผู้นำชุมชนและเจ้าหน้าที่สามารถรับเรื่อง ตรวจสอบ มอบหมายผู้รับผิดชอบ อัปเดตสถานะ บันทึกผลการดำเนินงาน และนำข้อมูลที่สะสมไปใช้วิเคราะห์ปัญหาในระยะยาว

ระบบนี้เป็นโปรเจ็กต์อิสระ มีชื่อ ข้อมูล ฐานข้อมูล บัญชีผู้ดูแล ไฟล์จัดเก็บ และระบบหลังบ้านแยกต่างหากทั้งหมด

# 2. วัตถุประสงค์

1. เพิ่มช่องทางแจ้งปัญหาที่ประชาชนเข้าถึงได้ง่าย
2. ลดการแจ้งปัญหาที่กระจัดกระจายผ่านโทรศัพท์ แชตส่วนตัว หรือกลุ่มสื่อสังคมออนไลน์
3. รองรับทั้งการแจ้งแบบระบุตัวตนและไม่ระบุตัวตน
4. ให้ผู้แจ้งสามารถติดตามผลการดำเนินงานได้
5. ช่วยให้เจ้าหน้าที่จัดลำดับ มอบหมาย และติดตามงานอย่างเป็นระบบ
6. เก็บประวัติการดำเนินงานและหลักฐานประกอบ
7. สร้างฐานข้อมูลปัญหาของชุมชนสำหรับวิเคราะห์ วางแผน และจัดทำโครงการ
8. ลดค่าใช้จ่ายประจำด้วยเทคโนโลยีที่มีโควตาใช้งานฟรี
9. ออกแบบระบบให้ดูแลต่อได้ง่ายและไม่ซับซ้อนเกินไป

# 7. สถานะของเรื่องแจ้ง

กำหนดสถานะหลักดังนี้:

| ค่าในระบบ | ชื่อที่แสดง | ความหมาย |
|---|---|---|
| `new` | รับเรื่องแล้ว | ระบบบันทึกเรื่องสำเร็จ |
| `reviewing` | กำลังตรวจสอบ | ผู้ดูแลกำลังตรวจสอบข้อมูล |
| `assigned` | มอบหมายแล้ว | มอบหมายให้เจ้าหน้าที่แล้ว |
| `in_progress` | กำลังดำเนินการ | เจ้าหน้าที่เริ่มดำเนินงาน |
| `waiting` | รอข้อมูลเพิ่มเติม | ต้องการข้อมูลหรือการประสานงานเพิ่ม |
| `resolved` | ดำเนินการแล้ว | แก้ไขหรือดำเนินการเสร็จ |
| `closed` | ปิดเรื่อง | ตรวจสอบและปิดกระบวนการแล้ว |
| `rejected` | ไม่รับดำเนินการ | เรื่องไม่อยู่ในขอบเขตหรือข้อมูลไม่เพียงพอ |
| `duplicate` | เรื่องซ้ำ | มีเรื่องเดิมอยู่แล้ว |

## 7.1 หลักการเปลี่ยนสถานะ

- ทุกการเปลี่ยนสถานะต้องบันทึกลง Timeline
- ต้องระบุผู้เปลี่ยนสถานะ
- ต้องบันทึกวันและเวลา
- สถานะ `rejected` ต้องมีเหตุผล
- สถานะ `resolved` ควรมีผลการดำเนินงาน
- สถานะ `closed` ใช้เมื่อจบกระบวนการแล้ว
- ห้ามลบประวัติสถานะเดิม

# 8. ระดับความสำคัญ

| ค่าในระบบ | ชื่อที่แสดง | ตัวอย่าง |
|---|---|---|
| `low` | ทั่วไป | ข้อเสนอแนะหรือปัญหาเล็กน้อย |
| `normal` | ปกติ | ปัญหาที่ควรดำเนินการตามลำดับ |
| `high` | เร่งด่วน | กระทบประชาชนหลายราย |
| `critical` | ฉุกเฉิน | เสี่ยงต่อชีวิตหรือความปลอดภัย |

> ระบบไม่ควรใช้แทนหมายเลขฉุกเฉิน ควรแสดงคำแนะนำและหมายเลขติดต่อฉุกเฉินอย่างชัดเจน

# 10. การแจ้งปัญหาของประชาชน

## 10.1 ข้อมูลในแบบฟอร์ม

### ข้อมูลปัญหา

- หมวดหมู่
- หัวข้อเรื่อง
- รายละเอียด
- วันที่พบปัญหา
- ระดับความเร่งด่วนตามที่ผู้แจ้งประเมิน

### ข้อมูลสถานที่

- ชื่อสถานที่
- หมู่ที่
- จุดสังเกต
- Latitude
- Longitude
- ลิงก์แผนที่

### ข้อมูลผู้แจ้ง

- รูปแบบการแจ้ง
  - ระบุตัวตน
  - ไม่ระบุตัวตน
- ชื่อ
- เบอร์โทร
- อีเมล (ไม่บังคับ)
- ช่องทางที่สะดวกให้ติดต่อกลับ

### รูปภาพ

- รูปภาพประกอบสูงสุด 3 รูปในรุ่นแรก
- รองรับ JPG, PNG และ WebP
- ลดขนาดด้านยาวไม่เกิน 1,600 พิกเซล
- บีบอัดก่อนอัปโหลด
- ขนาดหลังบีบอัดไม่ควรเกิน 1 MB ต่อรูป

### การยืนยัน

- ยืนยันว่าข้อมูลเป็นความจริง
- ยอมรับนโยบายความเป็นส่วนตัว
- ยินยอมให้ระบบใช้ข้อมูลเพื่อการประสานงาน

# 11. รหัสติดตามเรื่อง

## 11.1 รูปแบบรหัส

ตัวอย่าง:

```text
KPR-260625-A7F4
```

ความหมาย:

- `KPR` = Khaophang Report
- `260625` = วันที่สร้างแบบ YYMMDD
- `A7F4` = อักขระสุ่ม

## 11.2 ข้อกำหนด

- รหัสต้องไม่ซ้ำ
- ไม่ควรใช้เลขลำดับเพียงอย่างเดียว

# 19. การอัปโหลดภาพ

## 19.1 ขั้นตอน

```text
เลือกภาพ
  ↓
ตรวจชนิดไฟล์
  ↓
ลดขนาดด้วย Canvas
  ↓
บีบอัดเป็น JPEG หรือ WebP
  ↓
แปลงเป็น Base64
  ↓
ส่งไป Google Apps Script
  ↓
บันทึกใน Google Drive
  ↓
บันทึก File ID ใน Google Sheets
```

## 19.2 ข้อกำหนด

- สูงสุด 3 รูปต่อการแจ้งในรุ่นแรก
- ไม่เกิน 1 MB ต่อรูปหลังบีบอัด
- ความกว้างหรือความสูงไม่เกิน 1,600 พิกเซล
- ตรวจ MIME Type ที่ Backend
- ตั้งชื่อไฟล์ใหม่ ไม่ใช้ชื่อเดิมโดยตรง
- ป้องกันไฟล์อันตราย
- ไม่รับไฟล์ Script หรือ Executable
- จำกัดจำนวนคำขออัปโหลด

# 20. ระบบ Login และสิทธิ์

## 20.1 บทบาท

| Role | สิทธิ์ |
|---|---|
| `super_admin` | ทุกฟังก์ชัน |
| `admin` | จัดการเรื่อง หมวด และรายงาน |
| `officer` | ดูและอัปเดตเรื่องที่ได้รับมอบหมาย |
| `viewer` | ดู Dashboard และรายงานเท่านั้น |

## 20.2 ข้อกำหนดด้านรหัสผ่าน

- ห้ามเก็บรหัสผ่านแบบข้อความธรรมดา
- ต้องเก็บ Hash
- ควรใช้ Salt
- ความยาวขั้นต่ำ 8 ตัวอักษร
- ห้ามส่งรหัสผ่านกลับไป Frontend
- ไม่บันทึกรหัสผ่านใน Log

## 20.3 Session

- สร้าง Token แบบสุ่ม
- เก็บเฉพาะ Token Hash ในฐานข้อมูล
- Session มีเวลาหมดอายุ
- Logout ต้องยกเลิก Session
- ตรวจ Session ทุก Admin API
- สามารถบังคับออกจากระบบได้
- ไม่เก็บข้อมูลสิทธิ์สำคัญจาก Frontend เป็นแหล่งจริง

# 21. ความปลอดภัย

## 21.1 หลักการสำคัญ

- ไม่เชื่อข้อมูลจาก Frontend โดยอัตโนมัติ
- ตรวจสอบข้อมูลซ้ำที่ Backend
- จำกัดความยาวทุกช่อง
- ล้างและ Escape ข้อความก่อนแสดงผล
- หลีกเลี่ยง `innerHTML` กับข้อมูลผู้ใช้
- จำกัดชนิดและขนาดไฟล์
- ป้องกันคำขอซ้ำ
- ใช้ Lock Service ตอนสร้างรหัสหรือเขียนข้อมูลสำคัญ
- ใช้ Rate Limit แบบเหมาะสม
- เก็บ Secret ใน Script Properties
- ไม่ใส่ Spreadsheet ID, Folder ID หรือ Secret ใน Public Repository หากไม่จำเป็น
- บันทึก Activity Log
- ใช้ Soft Delete

## 21.2 Rate Limit เบื้องต้น

ตัวอย่าง:

- แจ้งเรื่อง: ไม่เกิน 5 ครั้งต่อ 10 นาทีต่ออุปกรณ์หรือคีย์ประกอบ
- ติดตามเรื่อง: ไม่เกิน 30 ครั้งต่อ 10 นาที
- Login: ไม่เกิน 5 ครั้งต่อ 15 นาที
- อัปโหลดภาพ: จำกัดตามจำนวนภาพต่อเรื่อง
- Admin API: ตรวจ Session ทุกครั้ง

> Apps Script ไม่สามารถระบุ IP ได้อย่างเชื่อถือได้ทุกกรณี จึงควรใช้หลายปัจจัยร่วมกัน เช่น Device Key, Session, เวลา และรูปแบบคำขอ

- เก็บเท่าที่จำเป็น
- แจ้งวัตถุประสงค์ก่อนส่งข้อมูล

- ไม่แสดงชื่อหรือเบอร์โทรในหน้าติดตาม

# 27. การจัดการ Error

กำหนด Error Code กลาง เช่น:

| Code | ความหมาย |
|---|---|
| `VALIDATION_ERROR` | ข้อมูลไม่ถูกต้อง |
| `UNAUTHORIZED` | ยังไม่ได้เข้าสู่ระบบ |
| `FORBIDDEN` | ไม่มีสิทธิ์ |
| `NOT_FOUND` | ไม่พบข้อมูล |
| `DUPLICATE` | ข้อมูลซ้ำ |
| `RATE_LIMITED` | ใช้งานถี่เกินไป |
| `FILE_TOO_LARGE` | ไฟล์ใหญ่เกินไป |
| `INVALID_FILE_TYPE` | ประเภทไฟล์ไม่รองรับ |
| `SESSION_EXPIRED` | Session หมดอายุ |
| `INTERNAL_ERROR` | ระบบขัดข้อง |

Frontend ต้องแปลง Error เป็นข้อความภาษาไทยที่เข้าใจง่าย

# 35. กฎการพัฒนา

14. ทุก Error ต้องมีข้อความที่ผู้ใช้เข้าใจได้
15. ต้องทดสอบบนมือถือจริงก่อน Deploy
16. Commit งานเป็นส่วนย่อยและเขียนข้อความให้ชัดเจน
17. ก่อนแก้โครงสร้าง Sheet ต้องสำรองข้อมูล
18. ไม่เปลี่ยนชื่อคอลัมน์โดยไม่แก้เอกสารและโค้ดทั้งหมด
